# PhoenixKit Locations — план доработки (иерархия площадок)

Дата: 2026-07-10. Статус: форк `timujinne/phoenix_kit_locations` (upstream `BeamLabEU/phoenix_kit_locations`) подключён в andi как path dep (Hex-строка `~> 0.2.0` закомментирована в mix.exs), v0.2.1. Связанные планы: `phoenix_kit_manufacturing/dev_docs/DEVELOPMENT_PLAN.md` (производство), `phoenix_kit_warehouse/dev_docs/DEVELOPMENT_PLAN.md` (склад).

## 0. Что уже есть в модуле

- **Location** (`phoenix_kit_locations`) — плоская карточка площадки: адрес (междунар. формат), контакты, статус, `features` map, мультиязычный `data`. Иерархии на уровне Location нет.
- **LocationType** (`phoenix_kit_location_types`) — пользовательские типы, many-to-many через `location_type_assignments`. В БД andi создан один тип: **Warehouse** («Physical warehouse / stock-keeping location»). Локации: Test, Main storage.
- **Space** (`phoenix_kit_location_spaces`) — **готовое дерево вложенных зон внутри локации**: обязательный FK на Location + self-ref `parent_uuid` (произвольная глубина), `position` + `reorder_siblings/4`, статусы, мультиязычный `data`. App-слой пока разрешает kinds `floor | room`; CHECK в БД (миграция ядра V122) уже допускает `hall, suite, section, zone, aisle, shelf, corner`.
- **Attachments** — файлы + заглавное изображение и у Location, и у каждого Space (folder_uuid/featured_image_uuid в `data`), мульти-ресурсный паттерн (несколько карточек файлов на одной странице).
- LiveViews: список локаций, форма локации, форма типа. **UI для Spaces ещё нет** — только контекст `Spaces` (list_tree, CRUD, reorder).

## 1. Архитектурное решение: два уровня иерархии

**Location = самостоятельная площадка** (здание/территория с адресом): склад, цех, офис. Типизируется через LocationType.

**Space = внутренняя структура площадки** (дерево): этаж → помещение → зона → участок → стеллаж. Типизируется через `kind`.

**Станки НЕ являются locations/spaces.** Станок — отдельная сущность в phoenix_kit_manufacturing (большой паспорт, файлы, справочники, виды операций) со ссылками `location_uuid` + `space_uuid` («стоит в цехе X, в зоне Y»). Location отвечает на вопрос «где», manufacturing — «что там стоит».

Обоснование: не трогаем схему Location (не нужен parent_uuid — дерево уже есть у Space), адрес/контакты остаются только у реальных зданий, а внутренние узлы остаются лёгкими.

## 2. Типы верхнего уровня (LocationType) — создать в справочнике

| Тип | Назначение |
|---|---|
| Warehouse | склад (уже создан) |
| Workshop / Цех | производственная площадка |
| Office / Офис | офисное здание |

Создаются через UI (Admin → Locations → Types), миграций не требуют. Одна площадка может совмещать типы (цех + склад) — many-to-many это уже позволяет.

## 3. Внутренние уровни (Space.kind) — расширить app-слой

Сейчас `@kinds ~w(floor room)`. Расширить до (все значения уже разрешены CHECK-констрейнтом БД, миграция не нужна):

| kind | Русское название | Применение |
|---|---|---|
| floor | Этаж | верхнее деление здания |
| room | Помещение | комната/кабинет |
| zone | Зона | производственная зона, зона хранения/приёмки/отгрузки |
| section | Участок | производственный участок (рабочий пост) — на него ссылаются участки/станки производства |
| aisle | Ряд/проход | адресное хранение на складе |
| shelf | Стеллаж/полка | адресное хранение на складе |

Задачи:
- `Space.@kinds` → полный список + gettext-лейблы (en/et/ru);
- правила вложенности (мягкие, на уровне подсказок UI, не констрейнтов): shelf под zone/aisle, section под floor/zone и т.п. — жёсткие правила пока не вводим;
- иконки/цвета по kind в UI дерева.

## 4. UI дерева Spaces (главная доработка)

В карточке локации — вкладка «Структура» (Structure):
- дерево с разворачиванием, добавление ребёнка любого kind, drag/кнопки порядка (`reorder_siblings`), rename, деактивация;
- карточка Space: описание, заметки, файлы (Attachments уже мульти-ресурсный — карточек файлов может быть несколько на странице);
- хлебная крошка полного пути («Main storage / Этаж 1 / Зона А / Стеллаж 3»).

## 5. API для других модулей

- `Paths.full_path(space_uuid)` → строка полного адреса (Location + цепочка Space) — для отображения в складских/производственных документах;
- **компонент-пикер** «выбрать место»: селект Location (с фильтром по типу) + дерево Space — переиспользуют склад (адрес хранения) и производство (местоположение станка/участка);
- лёгкие функции резолва: `get_location/1`, `get_space/1`, `list_locations(type: "Warehouse")` — уже есть или тривиальны.

## 7. Взаимосвязи с другими модулями

- **Склад**: `Stock.location_uuid` (сейчас мягкий UUID, single-warehouse) начинает указывать на реальные Location типа Warehouse → мульти-склад; фаза 2 — `space_uuid` в остатках/строках документов для адресного хранения. См. план склада §1–2.
- **Производство**: станок/участок ссылаются на `location_uuid` + `space_uuid`; kind `section` — «адрес» производственного участка. См. план производства §1–2.
- **Файлы**: инструкции/гарантии станков живут у станка (manufacturing), а не у Space; у Space — файлы самой зоны (планировки, схемы).

## 8. Процесс

- Работаем в форке (`/www/phoenix_kit_locations`, origin timujinne), PR в upstream BeamLabEU по мере готовности блоков.
- Новые значения kind укладываются в существующий CHECK; если понадобятся новые kinds сверх списка V122 или новые поля — PR в ядро phoenix_kit (versioned migration), как у склада.
- CHANGELOG/@version не трогаем (maintainer-owned).
